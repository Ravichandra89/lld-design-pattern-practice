// Designing Single Restaurant order system 

1. Features : 
    - Menu Management : Items, category, price, availability
    - Customer Workflow: view menu, add to cart, place order, payment.
    - Order Lifecycle: Place -> confirmed -> prepare -> ready -> picup/deliver -> completed / cancelled
    - Kitchen UI : latest orders, mark progress, update status
    - Notification: SMS/Push/Email for Order status
    - Payment: Accept payment (simulate payment gateways), Stripe, Paypal, Cash on delivery
    
    Additional not now to implement features: 
        - Coupons & Discounts
        - Table reservation & Dine restaurant
        - Analytics Dashboard
        - Admin Controls

2. Non - Functional Requirements: 
    - Availability : System shouldn't failure highly available
    - Extensible : Adding more features no effect (Low Coupling).
    - Correctness: no double order
    - Lower latency : Write/Read API's

3. High Level Components: 
    - Client Layer
    - API Layer - REST API's

    - Service Layer : 
        - Menu Service
        - Order Service
        - Cart Service
        - Payment Service
        - Notification Service
        - Kitchen Staff Service

    - Persistency :
        - Database - RDBMS
        - Cache - Static Database
        - Message Brokers - Kafka 

4. Data Modeling:

        - id : string
        - name: string (Veg/Non)

        - is_available : bool
        - quantity: int
        - updated_at 

    - MenuCategory: 
        - id : string
        - name: string
        - description: string
        - created_at : date

    - Customer: 
        - id : UUID
        - name: string
        - phone: string

    - customer_address:

        - address_line1
        - address_line2
        - city(string)
        - state (string)

    - Order :

5. API Design: 
    - Customer API'
        - GET /api/v1/menu - list menu Items
        - GET /api/v1/menu/{itemId} - get item Details
        - POST /api/v1/cart - add to cart
        - GET /api/v1/cart - Read cart
        - POST /api/v1/order - Create Order
        - GET /api/v1/{orderId} - Read order
        - GET /customer/{id}/orders - Show customer History

    - Kitchen Staff API: 
        - GET /api/v1/kitchen/orders?status=PLACED,CONFIRMED - show latest active orders
        - POST /api/v1/kitchen/order/{orderId}/status - Update order status
        - PATCH /api/v1/kitchen/item - update availability

    - Internal Payment : 
        - POST /api/v1/payment - make payment through gateways
        - POST /api/v1/notification - send notification

6. Workflow : 
    - placing an Order: 
    - kitchen picup order and prepare
    - payment & notification Management

7. Data Storage Choice: 
    - Primary DB: PostgreSQL - ACID
    - Cache: Redis
    - Message Brokers - Handling Kitchen events

8. Design Patterns: 
    - Singleton: MenuCatalog
    - Strategy: CreditCard, COD, Paypal
    - Observer: Notification System
